# Varlink IPC Interface

## Overview

avocadoctl exposes a Varlink IPC interface over a Unix socket, enabling programmatic access to all avocadoctl operations from any language. The `avocadoctl serve` subcommand runs a Varlink server with systemd socket activation support.

## Why Varlink over Unix socket (not a C shared library)

A C shared library was considered and rejected for several reasons:

**1. Rust has no stable ABI.** Rust's internal type representations (struct layout, vtable format, enum discriminants) can change between compiler versions. A `.so` built with rustc 1.78 may be incompatible with code expecting the layout from rustc 1.76. The only stable ABI Rust guarantees is the C ABI via `#[repr(C)]`, which means every public type, every error variant, every return struct must be manually annotated and frozen. This is a large ongoing maintenance burden.

**2. Version skew.** The avocadoctl binary on the runtime could be at a different version than a C library linked at build time. With a shared library, a struct gaining a new field or changing size is a silent ABI break -- callers read garbage memory or segfault. Mitigation crates like `abi_stable` and `stabby` exist but add significant complexity and still require careful version management.

**3. Privilege boundary.** avocadoctl's core operations (`systemd-sysext merge`, `systemd-confext merge`, `mount`, `depmod`, `modprobe`) require root. A C library linked into a caller's process means that process must run as root to perform these operations. If the caller doesn't need root otherwise, the library would need to internally communicate with a privileged daemon -- at which point it reinvents IPC but with an extra layer and all the ABI problems on top.

**4. State and lifecycle.** avocadoctl manages system-wide state (extension symlinks, loop mounts, systemd unit drop-ins). If multiple processes link the library and call merge/unmerge concurrently, there's no serialization. A daemon with a socket naturally serializes requests.

**5. Dependency chain.** The library would pull in the entire Rust runtime, TUF verification (`tough`), HTTP client (`ureq`), crypto (`ed25519-compact`, `sha2`), etc. into the caller's address space. Any security vulnerability in these dependencies requires rebuilding and redeploying all consumers -- not just updating avocadoctl.

### Why Varlink

Varlink is a typed IPC protocol using JSON messages over Unix sockets with a self-describing IDL. It addresses every concern above:

- **Version skew**: Built-in introspection (`org.varlink.service.GetInfo`) lets callers discover what the server supports at runtime. Adding optional fields is backward-compatible by design. No ABI -- just a wire protocol.
- **Privilege separation**: The daemon runs as root; callers connect over a Unix socket and don't need elevated privileges themselves.
- **Serialization**: Single daemon process handles all requests sequentially.
- **C support**: `sd-varlink` is built into libsystemd (available since v257, target is v258) -- zero extra dependencies. Alternatively, `vali` provides C code generation from IDL files for a fully typed struct-based experience.
- **systemd alignment**: systemd is actively migrating from D-Bus to Varlink. This follows the platform's direction.
- **Rust support**: The `varlink` crate (v13.0.0) provides sync server support with code generation from `.varlink` IDL files.

## Architecture

### Service layer

Business logic is extracted from CLI command handlers (`src/commands/*.rs`) into a reusable service layer (`src/service/*.rs`). Both the CLI and Varlink handlers call the same code. Service functions return `Result<T, AvocadoError>` instead of printing output and calling `process::exit(1)`.

Files:
- `src/service/mod.rs` -- module declarations
- `src/service/error.rs` -- unified `AvocadoError` enum
- `src/service/types.rs` -- shared response types
- `src/service/ext.rs` -- extension operations
- `src/service/runtime.rs` -- runtime operations
- `src/service/hitl.rs` -- hardware-in-the-loop operations
- `src/service/root_authority.rs` -- trust anchor operations

### Varlink interfaces

Four Varlink interfaces are defined as `.varlink` IDL files, with Rust code generated by `varlink_generator` at build time:

**org.avocado.Extensions** -- Extension management:
```varlink
interface org.avocado.Extensions

type Extension (
    name: string, version: ?string, path: string,
    isSysext: bool, isConfext: bool, isDirectory: bool
)

type ExtensionStatus (
    name: string, version: ?string,
    isSysext: bool, isConfext: bool, isMerged: bool,
    origin: ?string, imageId: ?string
)

method List() -> (extensions: []Extension)
method Merge() -> ()
method Unmerge(unmount: ?bool) -> ()
method Refresh() -> ()
method Enable(extensions: []string, osRelease: ?string) -> (enabled: int, failed: int)
method Disable(extensions: ?[]string, all: ?bool, osRelease: ?string) -> (disabled: int, failed: int)
method Status() -> ()

error ExtensionNotFound (name: string)
error MergeFailed (reason: string)
error UnmergeFailed (reason: string)
error ConfigurationError (message: string)
error CommandFailed (command: string, message: string)
```

**org.avocado.Runtimes** -- Runtime lifecycle:
```varlink
interface org.avocado.Runtimes

type RuntimeInfo (name: string, version: string)
type ManifestExtension (name: string, version: string, imageId: ?string)
type Runtime (
    id: string, manifestVersion: int, builtAt: string,
    runtime: RuntimeInfo, extensions: []ManifestExtension, active: bool
)

method List() -> (runtimes: []Runtime)
method AddFromUrl(url: string) -> ()
method AddFromManifest(manifestPath: string) -> ()
method Remove(id: string) -> ()
method Activate(id: string) -> ()
method Inspect(id: string) -> (runtime: Runtime)

error RuntimeNotFound (id: string)
error AmbiguousRuntimeId (id: string, candidates: []string)
error RemoveActiveRuntime ()
error StagingFailed (reason: string)
error UpdateFailed (reason: string)
```

**org.avocado.Hitl** -- Hardware-in-the-loop testing:
```varlink
interface org.avocado.Hitl

method Mount(serverIp: string, serverPort: ?string, extensions: []string) -> ()
method Unmount(extensions: []string) -> ()

error MountFailed (extension: string, reason: string)
error UnmountFailed (extension: string, reason: string)
```

**org.avocado.RootAuthority** -- Trust anchor information:
```varlink
interface org.avocado.RootAuthority

type TrustedKey (keyId: string, keyType: string, roles: []string)
type RootAuthorityInfo (version: int, expires: string, keys: []TrustedKey)

method Show() -> (authority: ?RootAuthorityInfo)

error NoRootAuthority ()
error ParseFailed (reason: string)
```

### Error mapping

Each `AvocadoError` variant maps 1:1 to a Varlink `error` declaration. For example, `AvocadoError::RuntimeNotFound { id }` becomes `org.avocado.Runtimes.RuntimeNotFound { "id": "..." }`.

### Server implementation

- `src/varlink_server.rs` -- implements the generated Varlink interface traits, maps `AvocadoError` to Varlink error replies
- `src/varlink/mod.rs` -- module declarations for generated code
- `src/main.rs` -- `serve` subcommand:
  ```
  avocadoctl serve [--address=unix:/run/avocado/avocadoctl.sock]
  ```

## systemd Integration

### Socket activation

The daemon is started on-demand via systemd socket activation when a connection arrives.

`systemd/avocadoctl.socket`:
```ini
[Unit]
Description=Avocado Control Varlink Socket

[Socket]
ListenStream=/run/avocado/avocadoctl.sock
FileDescriptorName=varlink
SocketMode=0660
SocketGroup=avocado
RemoveOnStop=yes

[Install]
WantedBy=sockets.target
```

`systemd/avocadoctl.service`:
```ini
[Unit]
Description=Avocado Control Daemon (Varlink)
Requires=avocadoctl.socket
After=avocadoctl.socket

[Service]
Type=simple
ExecStart=/usr/bin/avocadoctl serve
ProtectHome=yes
ProtectSystem=full
PrivateTmp=yes
NoNewPrivileges=yes

[Install]
WantedBy=multi-user.target
```

## Access Control

Unix socket permissions only -- no Polkit. The socket uses `SocketMode=0660` with `SocketGroup=avocado`, so any user in the `avocado` group can connect. Since avocadoctl operations (merge, mount, etc.) require root-level systemd calls, the daemon runs as root. The socket group controls who can connect.

## C Integration

While Varlink uses JSON on the wire, C code never does raw JSON string parsing. Both libraries below provide typed accessor APIs.

### Option A: sd-varlink from libsystemd (recommended)

Uses systemd's built-in Varlink client and structured JSON variant API. No extra dependencies -- `libsystemd` is already on every systemd v258 system. The `sd_json_variant` API provides typed field access (by name) on opaque objects, similar to GVariant or json-c.

```c
#include <systemd/sd-varlink.h>
#include <systemd/sd-json.h>

sd_varlink *vl = NULL;
sd_varlink_connect_address(&vl, "/run/avocado/avocadoctl.sock");

sd_json_variant *result = NULL;
sd_varlink_call(vl, "org.avocado.Extensions.List", NULL, &result);

// Walk a typed object tree -- no raw JSON string parsing
sd_json_variant *exts = sd_json_variant_by_key(result, "extensions");
for (size_t i = 0; i < sd_json_variant_elements(exts); i++) {
    sd_json_variant *ext = sd_json_variant_by_index(exts, i);
    const char *name = sd_json_variant_string(
        sd_json_variant_by_key(ext, "name"));
    bool is_sysext = sd_json_variant_boolean(
        sd_json_variant_by_key(ext, "isSysext"));
    // ...
}

sd_json_variant_unref(result);
sd_varlink_unref(vl);
```

Compile: `pkg-config --cflags --libs libsystemd`

### Option B: vali (standalone, with C code generation)

`vali` is a standalone C Varlink library by Simon Ser. Its distinguishing feature is a code generator that reads `.varlink` IDL files and produces native C structs and typed function signatures -- so the C code looks like calling a regular C library with no JSON concepts visible at all.

Given the IDL `method List() -> (extensions: []Extension)`, vali generates:

```c
// Auto-generated
struct avocado_Extension {
    char *name;
    char *version;   // NULL when absent
    char *path;
    bool isSysext;
    bool isConfext;
    bool isDirectory;
};

struct avocado_List_out {
    struct avocado_Extension *extensions;
    size_t extensions_len;
};

bool avocado_Extensions_List(struct vali_client *client,
    struct avocado_List_out *out, struct vali_error *err);
```

Usage:

```c
struct vali_client *client = vali_client_connect_unix("/run/avocado/avocadoctl.sock");

struct avocado_List_out out;
if (avocado_Extensions_List(client, &out, NULL)) {
    for (size_t i = 0; i < out.extensions_len; i++) {
        printf("%s\n", out.extensions[i].name);
    }
    avocado_List_out_finish(&out);
}
vali_client_destroy(client);
```

**Tradeoff**: vali is newer and less battle-tested than sd-varlink, but provides the most ergonomic C experience. It's an extra build dependency.

## Shipped artifacts

- **IDL files**: The `.varlink` IDL files are installed to `/usr/share/varlink/interfaces/`. These can be used with vali's code generator or referenced as documentation for sd-varlink accessor calls.
- **systemd units**: `avocadoctl.socket` and `avocadoctl.service` for socket-activated daemon.
